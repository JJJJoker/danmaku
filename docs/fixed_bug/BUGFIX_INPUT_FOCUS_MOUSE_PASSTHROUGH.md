# 输入框聚焦误关弹幕窗口鼠标穿透修复报告

- **日期**：2026-07-12
- **分支**：`dev`
- **问题反馈**：点击控制面板的「输入弹幕内容」/「语音弹幕」输入框后，弹幕层背后的其它应用/桌面点击无反应（用户在 Windows 上观察到，怀疑为遗留问题、可能双平台都有）
- **结果**：定位到 1 个根因，修复 1 项；客户端测试全绿、tsc 与基线比对无新增错误、build:vite 编译通过

## 总体结论

两个输入框挂着旧单窗口架构时代残留的 `onFocus`/`onBlur` 处理器，聚焦时通过 IPC 关闭的不是控制面板自己的鼠标穿透，而是**全屏透明置顶弹幕窗口**的穿透——这个视觉上不可见的窗口随即吞掉整屏鼠标事件，背后应用自然点不动。现双窗口架构下控制面板窗口本身常驻接收鼠标事件，这对处理器有害无用，直接删除即修复。

## 问题清单与修复

### Bug A — 输入框聚焦导致全屏弹幕窗口吞掉整屏点击

- **位置**：`src/renderer/components/ControlPanel.tsx`（弹幕输入框与语音弹幕输入框的 `onFocus`/`onBlur`，修复前位于 558-563、610-615 行）
- **问题**：完整链路为——
  1. 输入框 `onFocus` 调用 `window.electronAPI.setIgnoreMouseEvents(false)`；
  2. 该 IPC 在主进程 `src/main/main.ts` 的 `set-ignore-mouse-events` 处理器中**硬编码作用于 `mainWindow`（全屏透明置顶弹幕窗口）**，与调用方是哪个窗口无关；
  3. 于是点击输入框 → 铺满全屏、视觉不可见的弹幕窗口关闭鼠标穿透 → 吞掉整屏点击 → 背后应用点不动；
  4. `onBlur` 理论上会恢复穿透，但焦点切到其它应用（Alt-Tab / 点任务栏）时 input 不触发 DOM blur，穿透会一直关着，最长要等主进程 5 秒兜底定时器复位。

  这对处理器是旧单窗口架构（控制面板和弹幕层同窗）的残留：当时输入确实需要窗口临时接收鼠标。现双窗口架构下两窗口始终带 `?window=` 参数加载，控制面板窗口创建时即 `setIgnoreMouseEvents(false)` 常驻接收鼠标，输入框聚焦无需任何穿透操作。`RoomPanel.tsx` 中多处「操作后手动复位穿透，防止卡住」的零散补丁也是在给这套错误模式善后。该问题非近期回归——普通弹幕输入框的处理器自仓库最早提交即存在，语音输入框的是 v1.3.0 照抄引入。
- **修复**：删除两个输入框的 `onFocus`/`onBlur` 属性；并在 `src/main/main.ts` 的 `set-ignore-mouse-events` IPC 处理器上补注释，写明「无论哪个窗口调用，始终作用于弹幕窗口」以及输入框聚焦不应调用本 IPC 的原因，防止再犯。

### 有意保留（勿当遗漏改动）

- `DanmakuLayer.tsx` 拖拽调整弹幕区域时的穿透开关——弹幕窗口自己的合法用途；
- `RoomPanel.tsx` 与 `ControlPanel.tsx` 挂载/操作后的防御性穿透复位、主进程 5 秒兜底定时器——无害的安全网，继续作为最后防线；
- macOS 下 `focusin`/`focusout` → `setControlWindowLevel`——用于输入法候选框层级，与本 bug 无关；
- **未**把 IPC 改为按 sender 窗口路由：`ControlPanel.tsx`、`RoomPanel.tsx` 多处调用是有意从控制面板窗口操作弹幕窗口穿透，按 sender 路由会把它们全部改错。
